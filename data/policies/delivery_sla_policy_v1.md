# 物流时效与赔付政策 v1

## 1. 政策目的

本政策用于判断电商订单是否存在物流履约异常，以及是否可以进入延迟补偿或人工复核流程。它描述的是售后运营规则，不代表任何真实平台的正式赔付承诺。

## 2. 预计送达基准

每个订单都应有一个预计送达日期。业务判断应以平台在订单履约过程中承诺或展示给客户的预计送达日期为基准。

在当前 Olist 数据集中，该基准字段映射为 `order_estimated_delivery_date`。

## 3. 延迟送达判断

如果订单已经送达，且实际客户签收时间晚于预计送达日期超过 2 个自然日，可以进入延迟补偿判断。

在当前 Olist 数据集中，实际客户签收时间字段映射为 `order_delivered_customer_date`。判断逻辑为：

- `order_status` 等于 `delivered`
- `order_delivered_customer_date` 晚于 `order_estimated_delivery_date`
- 两者差值大于 2 个自然日

满足以上条件只表示可以进入补偿候选流程，不表示 Agent 可以直接承诺赔付。

## 4. 未送达履约异常

如果订单状态不是 `delivered`，且已经超过预计送达日期 5 个自然日仍无送达记录，应生成履约异常工单，由人工判断是否需要催单、退款、补偿或联系卖家。

当前数据集没有实时物流轨迹，因此只能使用订单状态与预计送达日期做近似判断。真实系统应接入物流轨迹、承运商状态和客服记录。

## 5. 补偿边界

延迟补偿建议必须同时参考订单金额、客户沟通历史、卖家责任、是否已有退款或补偿记录，以及平台风险策略。

当证据不足、金额较高、客户诉求复杂、或需要创建工单时，必须进入人工审批。

## 6. Agent 执行限制

Agent 可以输出“初步满足延迟补偿复核条件”“建议创建待审批工单”等结构化判断。

Agent 不得直接承诺现金赔付、优惠券赔付、退款成功、或任何已经执行的资金动作。
